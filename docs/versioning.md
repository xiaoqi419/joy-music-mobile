# 版本号标识规范（悦音）

本文档用于统一项目版本号含义、发布标签和升级策略。

## 1. 基本格式

项目使用语义化版本（SemVer）：

`主版本号.次版本号.修订号`

示例：`1.0.1`

## 2. 每一位代表什么

以 `1.0.1` 为例：

1. `1`（主版本号 Major）  
   表示大版本。出现不兼容变更、架构升级、重大功能重构时递增。

2. `0`（次版本号 Minor）  
   表示功能版本。新增功能、增强体验但保持兼容时递增。

3. `1`（修订号 Patch）  
   表示补丁版本。修复 Bug、优化细节、不改功能边界时递增。

## 3. 递增规则

1. 发大版本：`1.4.6 -> 2.0.0`
2. 发功能版本：`1.4.6 -> 1.5.0`
3. 发修复版本：`1.4.6 -> 1.4.7`

每次只递增一个维度，右侧维度清零（Major/Minor 场景）。

## 4. 预发布版本（可选）

预发布建议使用：

1. `1.0.1-beta.1`：测试版本
2. `1.0.1-rc.1`：候选发布版本

建议流程：

1. `1.0.1-beta.1`
2. `1.0.1-beta.2`
3. `1.0.1-rc.1`
4. `1.0.1`（正式）

## 5. Git Tag 规范

CI 发布使用 `v` 前缀 tag：

1. 正式版：`v1.0.1`
2. 预发布：`v1.0.1-beta.1`

注意：

1. `Release Tag` 工作流输入版本号时填写 `1.0.1` 或 `1.0.1-beta.1`（不带 `v`）
2. 工作流会自动创建 `v<version>` tag

## 6. 与项目配置的联动

执行下面命令会同步版本号到项目关键文件：

```bash
npm run release:version -- 1.0.1
```

会同步到：

1. `package.json -> version`
2. `app.json -> expo.version`
3. `app.json -> expo.ios.buildNumber`
4. `src/config/index.ts -> appConfig.version`

## 7. 升级策略建议

1. `Patch`：默认平滑升级
2. `Minor`：建议升级（新功能）
3. `Major`：建议在更新提示中强调“可能存在行为变化”

## 8. 快速参考

1. 修一个线上 bug：升 `Patch`
2. 新增一个完整模块：升 `Minor`
3. 改了核心架构或不兼容行为：升 `Major`
